# GLOSSARY.md

This file is the living glossary for Portfolio X-Ray & Optimization Terminal / Portfolio MRI.

It keeps project terminology consistent across documentation, specs, code comments, reports, outputs, and future UI/API surfaces. It does not replace formulas, policy rules, data rules, or detailed specs.

Terms can be added, changed, renamed, or removed as the project vocabulary evolves.

## Purpose

- Keep important terms defined consistently.
- Reduce ambiguity between docs, specs, code, reports, and generated outputs.
- Make acronyms and project-specific concepts easier to understand.
- Link short definitions to the canonical source of truth when details live elsewhere.

## What Belongs Here

- Project-specific terms and acronyms.
- Financial, portfolio, data, stress, factor, optimizer, output, workflow, and documentation-governance terms used across multiple files.
- Short definitions that help prevent confusion.
- Links to canonical specs when the term has detailed formulas or behavior rules.

## What Does Not Belong Here

- Long formulas; put those in the owning spec.
- Full implementation contracts; put those in `SPEC.md` or `docs/specs/*.md`.
- Long methodology explanations; link to the relevant spec or ExecPlan.
- Product ideas that are not terms.

## Lifecycle

Terminology is dynamic.

- Add a term when it appears repeatedly or could be misunderstood.
- Update a term when its meaning changes in specs, code, reports, outputs, or UI/API language.
- Remove a term when it is obsolete and no longer used.
- Rename a term only after checking for stale references across docs and code.
- Keep definitions short; link to the canonical spec for details.

## Entry Format

Use this format for new entries:

```markdown
Term heading

- Definition: Short definition in one or two sentences.
- Area: metrics | data | optimizer | stress | factor_macro | reporting | outputs | workflow | docs | product
- Canonical source: Link to the owning spec or document.
- Notes: Optional short clarification, if needed.
```

- Definition: The currency in which portfolio returns, risk, and reporting are evaluated.
- Area: data

### FX conversion

- Definition: Conversion of asset price or return series into the investor currency before portfolio return calculations.
- Area: data
- Canonical source: [DATA.md](DATA.md)

### Return panel

- Definition: The aligned table of asset returns used by metrics, optimization, backtests, stress, and reports.
- Area: data
- Canonical source: [DATA.md](DATA.md)

### Returns frequency

- Definition: The configured return interval used by shared analytics, such as monthly, weekly, or daily where supported.
- Area: data
- Canonical source: [DATA.md](DATA.md)

### Risk-free rate

- Definition: The rate series used for excess-return metrics and related calculations where required.
- Area: metrics
- Canonical source: [docs/specs/metrics_specification.md](docs/specs/metrics_specification.md)

### Benchmark

- Definition: The reference asset or index used for comparison, beta, relative diagnostics, and reporting context.
- Area: metrics
- Canonical source: [SPEC.md](SPEC.md)

### Dynamic NaN-safe backtest

- Definition: Backtest mode that handles assets with different start dates and missing return observations without forcing a static full-history universe.
- Area: data
- Canonical source: [docs/specs/data_policy_spec.md](docs/specs/data_policy_spec.md)

### Young ETF

- Definition: An ETF or asset with limited return history that requires special handling in data, covariance, and optimization logic.
- Area: data
- Canonical source: [docs/specs/data_policy_spec.md](docs/specs/data_policy_spec.md)

### Portfolio weights

- Definition: Asset allocation weights used for portfolio analytics and reporting; policy weights are generated by the optimizer workflow.
- Area: optimizer
- Canonical source: [docs/specs/portfolio_construction_policy.md](docs/specs/portfolio_construction_policy.md)

### Analysis mode

- Definition: Config setting that defines whether the run builds policy weights from a ticker universe or diagnoses an existing fixed-weight portfolio.
- Area: input and assumptions
- Canonical source: [docs/specs/input_assumptions_spec.md](docs/specs/input_assumptions_spec.md)

### Analysis setup

- Definition: The resolved runtime contract for portfolio input, analysis portfolio, mandate, assumptions, and validation before downstream analytics consume the setup.
- Area: input and assumptions
- Canonical source: [docs/specs/input_assumptions_spec.md](docs/specs/input_assumptions_spec.md)

### Analysis subject

- Definition: The `analysis_subject` portfolio diagnosed first in the portfolio-first review workflow; it can be a user current portfolio, a user model portfolio, or a universe baseline and is resolved into `analysis_setup.analysis_subject`.
- Area: workflow
- Canonical source: [docs/specs/portfolio_review_workflow_spec.md](docs/specs/portfolio_review_workflow_spec.md)

### Input assumptions

- Definition: Exported/reporting view projected from `analysis_setup`; it is used for reproducibility and display, not as a separate business-logic source.
- Area: input and assumptions
- Canonical source: [docs/specs/input_assumptions_spec.md](docs/specs/input_assumptions_spec.md)

### Current weights

- Definition: User-supplied existing-portfolio weights used for diagnostics when `analysis_mode` is `analyze_current_weights`; they are not generated policy weights.
- Area: input and assumptions
- Canonical source: [docs/specs/input_assumptions_spec.md](docs/specs/input_assumptions_spec.md)

### Policy portfolio

- Definition: A portfolio produced by the legacy policy optimization workflow; it is not the default starting portfolio in the portfolio-first workflow.
- Area: optimizer
- Canonical source: [docs/specs/portfolio_construction_policy.md](docs/specs/portfolio_construction_policy.md)

### Candidate portfolio

- Definition: A comparison portfolio generated by a benchmark or robust script after weights are fixed; it does not automatically replace the policy portfolio.

- Canonical source: [docs/specs/candidate_portfolios_spec.md](docs/specs/candidate_portfolios_spec.md)

### Equal Weight

- Definition: Candidate portfolio construction that assigns equal weight across eligible assets or, in a separate variant, equal budget by asset class and then by asset.
- Area: optimizer
- Canonical source: [docs/specs/candidate_portfolios_spec.md](docs/specs/candidate_portfolios_spec.md)

### Risk Parity

- Definition: Candidate portfolio construction that targets balanced risk contribution rather than equal capital weight.
- Area: optimizer
- Canonical source: [docs/specs/candidate_portfolios_spec.md](docs/specs/candidate_portfolios_spec.md)

### Hierarchical Risk Parity

- Definition: Candidate portfolio construction using hierarchical clustering and recursive allocation on covariance/correlation structure.

- Canonical source: [docs/specs/candidate_portfolios_spec.md](docs/specs/candidate_portfolios_spec.md)

### Minimum Variance

- Definition: Candidate portfolio construction that seeks the lowest portfolio variance under the applicable constraint set for the variant.

- Canonical source: [docs/specs/candidate_portfolios_spec.md](docs/specs/candidate_portfolios_spec.md)

### Maximum Diversification

- Definition: Candidate portfolio construction that maximizes diversification ratio under the applicable constraint set for the variant.

- Canonical source: [docs/specs/candidate_portfolios_spec.md](docs/specs/candidate_portfolios_spec.md)

### Blocks 1–5 deliverable

- Definition: The diagnostic and candidate-prep scope covered by Blocks 1–5 audits and walkthroughs: input and assumptions, Portfolio X-Ray, Stress Test Lab, Candidate Portfolio Factory, and optimization-engine / candidate-generation readiness. It ends before Selection Engine, Action Plan, Monitoring, and Decision Journal are interpreted as the audit subject.

- Definition: The V1 generated decision-support bundle written after `candidate_comparison.json`: robustness scorecard, Portfolio Health Score, selection/no-trade, action plan, monitoring diff, decision journal, and decision-package summary (plus optional PDF when export is requested).

### Candidate factory run evidence

- Definition: Orchestration record in `candidate_factory_run.json` describing the **last** factory execution: profile (`core_v1` / `default_v1`), per-step status (`succeeded`, `skipped_existing`, …), timing, and pointers to comparison outputs.
- Area: outputs
- Canonical source: [docs/specs/candidate_factory_spec.md](docs/specs/candidate_factory_spec.md)
- Notes: Describes what the factory **last orchestrated**, not every row scored in comparison. A `core_v1` run with reuse does not prove the full `default_v1` menu was rebuilt.

### Candidate comparison evidence

- Definition: Aggregated diagnostic table in `candidate_comparison.json` built by scanning **existing** per-candidate artifacts on disk, annotated with `candidate_menu` and `review_bundle_context`.
- Area: outputs
- Canonical source: [docs/specs/candidate_comparison_spec.md](docs/specs/candidate_comparison_spec.md)
- Notes: Row set can be **wider** than the last factory run when snapshots were reused or prior full-menu folders remain. Read `candidate_menu.is_partial_menu` and `factory_evidence_status` before treating rankings as full-menu proof.

## Update Rules

- If a terminology change affects user-facing docs or reports, update [README.md](README.md), [OUTPUTS.md](OUTPUTS.md), or the relevant report spec.
- If a terminology decision is important, record the rationale in [DECISIONS.md](DECISIONS.md).
