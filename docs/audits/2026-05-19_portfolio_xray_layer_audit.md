Date: 2026-05-19

Status: Active input for [Portfolio X-Ray Diagnostics Deepening Plan](../exec_plans/2026-05-19_portfolio_xray_diagnostics_deepening_plan.md).

Scope: current-portfolio diagnostics only. This audit excludes candidate selection, candidate construction, optimizer policy, and advanced UI work except where they consume or present Portfolio X-Ray output.

Primary evidence reviewed:

This file records the audit conclusions used to create the next implementation plan. It is evidence and planning memory; it does not override current code or canonical specs.

## Executive Summary

The Portfolio X-Ray layer exists, but it is not yet a fully reliable diagnostic product. It is best described as a structured aggregator over existing metrics, stress, factor, and snapshot outputs. It already has a useful seven-section `portfolio_xray.json` surface, but several sections are shallow, some evidence wiring is incomplete, and report presentation is not yet advisor/investor friendly.

Estimated maturity:

- Technical existence: 55-60% of the intended layer.
- User-facing diagnostic usefulness: 45-50% of the intended layer.

Strongest areas:

- Portfolio metrics and stress/factor infrastructure are already broad.
- `portfolio_xray.json` already separates diagnostic sections and carries a diagnostic-only disclaimer.
- Portfolio-first `analysis_subject` materialization gives the right workflow location for current-portfolio diagnosis before candidates.

Weakest areas:

- Data-cutoff trust needs a P0 pass. Some generated diagnostic consumers can disclose or use dates after `analysis_end`.
- Risk Budget View is misleading because the X-Ray reads top-5 risk contribution evidence from `snapshot_10y.json`, while full `rc_vol_10y.csv` evidence exists.
- Kalman factor beta evidence is not surfaced because X-Ray reads stale field names.
- VaR / ES methodology is inconsistent with the metrics spec and needs either code alignment or explicit spec correction.
- Hidden Risk Detector, Portfolio Archetype, and Weakness Map are present but not yet decision-useful enough for the target product concept.
- HTML/report/PDF surfaces expose too much raw X-Ray structure instead of clear diagnostic tables and explanations.

## Sub-Block Audit

### 2.1 Asset Allocation

Purpose: answer what the portfolio owns, where capital is allocated, and whether visible concentration exists before any optimization comparison.

Current implementation: partially implemented. `portfolio_xray.json.asset_allocation` exists and is generated by `src/portfolio_xray.py` from weights, metadata, and snapshot/report context. The output is part of the portfolio-first `analysis_subject` workflow through `run_report.py --materialize-analysis-subject`.

Inputs: portfolio weights, ticker metadata, taxonomy/asset metadata where available, and generated snapshot context. It can support asset-level and some category views, but taxonomy completeness is uneven and V1 taxonomy is annotation-only.

Current coverage:

- Asset-level weights: implemented.
- Asset class: partially implemented through metadata/taxonomy.
- Region, currency, sector, risk bucket: incomplete or dependent on metadata coverage.
- ETF/stock taxonomy: partially available, not yet a complete user-facing allocation taxonomy.
- Concentration: basic weight concentration is present or derivable, but not yet a full concentration diagnostic.

Quality classification: partially implemented.

Key weaknesses:

- Missing or inconsistent taxonomy fields can make allocation views incomplete.
- There is no strong confidence/warning policy for partial taxonomy coverage.
- Concentration is not yet clearly separated into capital concentration, issuer/fund concentration, sector concentration, and risk concentration.

Improvement route: Sessions 04 and 08 should deepen concentration metrics and presentation; taxonomy expansion can be a later follow-up unless needed for hidden-risk flags.

### 2.2 Portfolio Metrics / Risk Diagnostics

Purpose: answer how the current portfolio has behaved historically, how much risk it took, how severe losses were, and whether risk-adjusted returns were acceptable before considering alternatives.

Current implementation: implemented but incomplete. `portfolio_xray.json.risk_diagnostics` aggregates metrics from the existing monthly report/snapshot flow and related analytics. Core metrics such as CAGR, volatility, Sharpe, Sortino, max drawdown, beta, and recovery-style diagnostics are present in the broader system.

Inputs: portfolio returns, adjusted prices, FX-adjusted returns, benchmark returns, risk-free inputs, and generated snapshots. Main portfolio metrics are monthly by default. Some daily diagnostics exist in separate paths, but the X-Ray metric contract does not yet consistently expose frequency and method metadata.

Key metric concerns:

- CAGR, volatility, Sharpe, Sortino, max drawdown, time to recovery: generally aligned with the metrics infrastructure, but output should disclose frequency/window/observation count.
- VaR / ES: P0 methodology mismatch. Current base tail metrics appear monthly in the X-Ray/snapshot path, while the metrics specification describes daily historical VaR/ES. This can mislead users about the horizon of the tail loss number.
- Beta and Treynor: beta exists in the metrics path; Treynor presence should be verified in code before expanding claims. Benchmark identity and frequency must be disclosed.
- Skewness/kurtosis: asset-level evidence exists in some outputs, but portfolio-level X-Ray coverage is missing.
- Rolling metrics and downside/upside beta: missing from X-Ray.
- Liquidity diagnostics: not yet a real portfolio-level diagnostic block.

Quality classification: implemented but shallow.

Key weaknesses:

- Frequency mixing is not visible enough.
- Tail-risk methodology is inconsistent with the current spec.
- Quality metadata such as `n_obs`, benchmark, risk-free source, and window is not consistently surfaced.

Improvement route: Sessions 03 and 04.

### 2.3 Factor Exposure / Factor Sensitivity

Purpose: answer which market factors the current portfolio is implicitly sensitive to, including equity, rates, inflation, credit, USD, commodities, volatility, and growth-style proxies.

Current implementation: partially implemented. The system has factor, macro, PCA, and stress artifacts. `portfolio_xray.json.factor_exposure` summarizes some of that evidence, but it is currently more of a supporting diagnostic summary than a complete factor sensitivity product.

Inputs: portfolio returns, benchmark/factor returns, stress report outputs, macro/factor data, PCA summaries, and generated factor beta outputs.

Current coverage:

- Equity/rates/inflation/credit/USD/commodities/volatility/growth proxies: partially available through factor/stress infrastructure, depending on configured factors and generated artifacts.
- PCA/common-factor evidence: available in supporting artifacts but not fully integrated as a user-facing X-Ray diagnosis.
- Kalman factor betas: generated in `stress_report.factor_betas_kalman.latest`, but X-Ray currently looks for stale names such as `latest_betas_capped` / `latest_betas`, so `kalman_current_beta` can be null even when evidence exists.

Quality classification: partially implemented.

Key weaknesses:

- Factor diagnostics are not yet consistently separated into current exposure, sensitivity method, confidence, and missing evidence.
- Kalman mapping bug hides available evidence.
- Factor labels may look more complete than the underlying data coverage supports.

Improvement route: Sessions 02, 04, and 05.

### 2.4 Hidden Exposure / Hidden Risk Detector

Purpose: answer what risk is not obvious from headline weights: hidden equity beta, duration concentration, credit concentration, liquidity concentration, correlation concentration, weak hedge behavior, tail risk, and macro/factor dependency.

Current implementation: technically present but shallow. `portfolio_xray.json.hidden_risk_detector` exists and uses rule-based flags over existing diagnostics, but it does not yet provide enough balanced evidence, non-flags, confidence, or category coverage to be decision-useful.

Inputs: weights, correlations, beta/factor evidence, PCA/stress diagnostics, risk metrics, and taxonomy metadata where available.

Current coverage:

- Hidden equity beta: partially supported by beta/factor evidence.
- Duration/credit concentration: partially supportable through asset taxonomy and factor/stress evidence, but not yet robustly explicit.
- Liquidity concentration: not a real diagnostic.
- Correlation/common-factor concentration: partially supportable from PCA/correlation evidence.
- Weak hedge behavior: not explicit enough.
- Tail risk: not explicit enough, and VaR/ES methodology needs fixing first.
- Hidden macro/factor dependency: partially supportable from factor/stress outputs.

Quality classification: technically present but not decision-useful enough.

Key weaknesses:

- Flags can look like definitive findings even when coverage is partial.
- Critical categories do not consistently show "not flagged / below threshold" evidence.
- No section-level confidence or evidence count.

Improvement route: Session 05 after metric/factor evidence wiring is fixed.

### 2.5 Portfolio Archetype Classification

Purpose: answer what kind of portfolio this is in plain language, with evidence and caveats: equity-growth, balanced/60-40-like, duration-heavy, credit-carry, inflation-sensitive, pseudo-diversified, defensive, or concentrated-risk.

Current implementation: technically present but fragile. `portfolio_xray.json.portfolio_archetype` provides a simple label and confidence, but the current method is too coarse for advisor/client use.

Inputs: allocation, risk metrics, factor exposure, stress behavior, and hidden-risk evidence.

Observed issue: a portfolio can be labeled "Inflation-sensitive" with high confidence while also showing high inflation/rates vulnerability. That contradiction needs to be represented as conflicting evidence, not hidden by a single label.

Quality classification: technically present but not decision-useful.

Key weaknesses:

- Archetype rules are not yet specified as a scorecard.
- Positive evidence, negative evidence, conflicts, and confidence are not explicit enough.
- The label can create false confidence.

Improvement route: Session 07, after Weakness Map V2 provides better regime evidence.

### 2.6 Risk Budget View

Purpose: answer whether capital weights match actual risk contribution, stress loss contribution, and factor risk contribution.

Current implementation: partially implemented with a P0 evidence-completeness bug. The risk contribution infrastructure writes full `results_csv/rc_vol_10y.csv`, but X-Ray currently relies on `snapshot_10y.json.RC_asset`, which is a top-5 display subset. As a result, positive-weight assets can appear with `RC_vol = null` in `portfolio_xray.json.risk_budget_view` even when full RC evidence exists.

Inputs: portfolio weights, covariance/risk contribution outputs, stress contribution outputs, and snapshot summaries.

Current coverage:

- Capital weight: implemented.
- Volatility risk contribution: implemented in data, incompletely surfaced in X-Ray.
- Stress loss contribution: partly available through stress outputs, not yet a full risk-budget view.
- Factor risk contribution: not a full user-facing risk budget.
- Drawdown contribution: not present.

Quality classification: broken/unstable for current X-Ray presentation, despite useful underlying data.

Key weaknesses:

- Top-5 snapshot evidence is used where full risk-budget evidence is required.
- Null values can be misread as missing risk or unavailable asset risk.
- Stress/factor contributions are not yet integrated into one risk budget view.

Improvement route: Session 02 is P0.

### 2.7 Portfolio Weakness Map

Purpose: answer where the current portfolio is vulnerable by scenario or regime: recession, inflation shock, rates-up shock, credit spread widening, liquidity shock, USD shock, equity crash, commodity shock, volatility spike, and crypto shock when relevant.

Current implementation: present but shallow. The system has stress outputs and a `weakness_map` section, but the map is closer to a severity list than a full vulnerability diagnostic.

Inputs: stress scenario analytics, scenario library outputs, factor evidence, taxonomy/asset relevance, and portfolio stress loss drivers.

Current coverage:

- Recession, inflation, rates-up, equity crash, and related macro shocks: partially represented through stress/scenario outputs.
- Credit, liquidity, USD, commodity, volatility: partially supportable, but not consistently mapped with evidence and drivers.
- Crypto shock: should be conditional on relevant holdings/taxonomy; not a general always-on row.

Quality classification: implemented but shallow.

Key weaknesses:

- Low severity can be misread as "risk does not exist" instead of "limited adverse evidence under current data."
- Exposure presence, adverse evidence, severity, and confidence are not separated.
- Top asset and factor loss drivers are not consistently surfaced.

Improvement route: Session 06.

## Cross-Block Issues

- Data cutoff consistency: diagnostic consumers must use rows `<= analysis_end`; raw cached panels may contain later incomplete periods but generated diagnostics must not claim them as analysis-effective.
- Frequency consistency: monthly, daily, and scenario outputs are mixed; X-Ray must disclose frequency/method/window for each metric.
- Metadata trust: taxonomy and asset metadata are incomplete for some allocation and hidden-risk diagnostics.
- Evidence completeness: X-Ray must read full diagnostic evidence where available, not display-oriented top-N snapshots.
- Generated artifact staleness: candidate and report outputs can be stale unless analysis metadata is checked.
- Weak commentary: the JSON structure exists, but user-facing report surfaces do not yet explain why findings matter.
- Legacy dependencies: some report paths still carry legacy assumptions or field names.
- False confidence risk: archetype labels, hidden-risk flags, and low weakness severity can overstate certainty.

## P0 / P1 / P2 Plan

P0:

- Fix analysis-effective date cutoff across diagnostic consumers.
- Fix Risk Budget View to use full risk contribution evidence.
- Fix Kalman factor beta mapping.
- Align VaR / ES methodology with the metrics spec or explicitly change the spec.

P1:

- Add missing portfolio metrics: skewness, kurtosis, downside/upside beta, rolling beta/correlation, and quality metadata.
- Build Hidden Risk Detector V2 with positive flags, non-flags, confidence, and evidence counts.
- Build Weakness Map V2 with exposure, adverse evidence, severity, confidence, and drivers.
- Rework Portfolio Archetype as an evidence scorecard with conflict handling.

P2:

- Productize report/HTML/PDF X-Ray presentation.
- Improve generated-output QA for X-Ray language.
- After diagnostic trust fixes, return to operational portfolio-first review modes.

## Final Verdict

About 55-60% of the intended Portfolio X-Ray layer exists today at a technical level, but only about 45-50% is ready as a user-facing diagnostic layer.

Strongest sub-block: Portfolio Metrics / Risk Diagnostics, because the broader metrics infrastructure is already mature.

Weakest sub-block: Portfolio Archetype Classification, because the current label can hide conflicting evidence and create false confidence.

First improvement: fix P0 trust issues in Sessions 01-03 before adding deeper diagnostics or report polish.

Recommendation: deepen the X-Ray layer before moving further into UI or advanced optimization.
