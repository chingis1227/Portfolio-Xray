# Portfolio X-Ray Diagnostics Specification

Status: source-of-truth for section contracts, thresholds, and JSON fields (Block 2).

Layer navigation (sub-blocks 2.1–2.7, code map, orchestration):
[portfolio_xray_layer_spec.md](portfolio_xray_layer_spec.md).

This spec defines the intended contract for the current-portfolio diagnostic layer. It is diagnostic-only: it describes how the system should explain an existing `analysis_subject` before any candidate comparison, optimization decision, or action plan.

## Scope

The Portfolio X-Ray layer diagnoses the current `analysis_subject` in the portfolio-first workflow:

```text
analysis_subject
-> subject diagnostics / Portfolio X-Ray
-> candidate generation and comparison
-> decision package
```

The layer is generated by the report path and written as `portfolio_xray.json` under the active output folder, including `{output_dir_final}/analysis_subject/` when `run_report.py --materialize-analysis-subject` is used.

The layer defines seven diagnostic sub-blocks. **Current Core MVP (product-facing Block 2)** includes
**2.1–2.6** (top-level `block_2_1_*` … `block_2_6_portfolio_weakness_map` on `portfolio_xray.json`).
**2.7** remains an **advanced / backlog / legacy-section** diagnostic: it may still populate
`sections.*` for backward compatibility and report formatters, but must not be promoted as a Core MVP
product surface and must not gain `block_2_7_*` modules without an explicit spec migration.
**Portfolio Archetype** must not use `block_2_*` product keys (forbidden: `block_2_5_portfolio_archetype`,
`block_2_6_portfolio_archetype`, `block_2_7_portfolio_archetype`).

| Sub-block | Core MVP product contract | Legacy `sections` key (compatibility) |
| --- | --- | --- |
| 2.1 Asset Allocation | **Yes** — `block_2_1_asset_allocation` | `asset_allocation` |
| 2.2 Portfolio Metrics / Risk Diagnostics | **Yes** — `block_2_2_portfolio_metrics` | `risk_diagnostics` |
| 2.3 Factor Exposure / Factor Sensitivity | **Yes** — `block_2_3_factor_exposure` | `factor_exposure` |
| 2.4 Hidden Exposure / Hidden Risk Detector | **Yes** — `block_2_4_hidden_exposure` | `hidden_risk_detector` |
| 2.5 Risk Budget View | **Yes** — `block_2_5_risk_budget_view` (§2.5.1) | `risk_budget_view` |
| 2.6 Portfolio Weakness Map | **Yes** — `block_2_6_portfolio_weakness_map` (§2.6.1) | `weakness_map` |
| 2.7 Portfolio Archetype Classification | **No** — advanced / legacy-section | `portfolio_archetype` |

## Non-Goals

Portfolio X-Ray must not:

- optimize the portfolio
- select candidates
- change current or generated weights
- change mandate gates, stress pass/fail status, or release decisions
- produce binding investment recommendations
- replace `candidate_comparison.json`, `selection_decision.json`, `action_plan.json`, or `portfolio_health_score.json`

Diagnostic statements may identify risks, gaps, evidence, uncertainty, and possible review areas. They must not be worded as binding buy/sell/hold instructions.

## Primary Artifacts

Machine-readable output:

- `portfolio_xray.json`

Current and target report consumers:

- `report.txt`
- `commentary.txt`
- generated HTML report surfaces
- generated PDF-style report surfaces

Generated report surfaces should render the X-Ray as structured sections and tables. `portfolio_xray.json` remains the canonical generated source for X-Ray rendering.

  "thresholds": {},

  "block_2_2_portfolio_metrics": {},
  "block_2_3_factor_exposure": {},
  "block_2_4_hidden_exposure": {},
  "block_2_5_risk_budget_view": {},
  "sections": {},
  "legacy_summary": {}

The current implementation uses `version: portfolio_xray_v2`. `block_2_1_asset_allocation` is the product-facing Block 2.1 contract (§2.1.1); it is populated on portfolio-first materialize paths when `build_portfolio_xray_v2` runs (ExecPlan closed Session 08, 2026-05-26). Older on-disk `portfolio_xray.json` files may lack the key until re-materialized.

`block_2_2_portfolio_metrics` is the product-facing Block 2.2 contract (§2.2.1); populated when `build_portfolio_xray_v2` runs (builder Session 03, 2026-05-26). Older on-disk files may omit the key until re-materialized. Legacy `sections.risk_diagnostics` remains for golden tests and formatters.

`block_2_3_factor_exposure` is the product-facing Block 2.3 contract (§2.3.1); populated when `build_portfolio_xray_v2` runs. It is an adapter over `stress_report` factor diagnostics and must not trigger OLS, Kalman, variance-decomposition, data-loading, candidate, or Stress Lab calculations. Older on-disk files may omit the key until re-materialized. Legacy `sections.factor_exposure` remains for golden tests and formatters.

`block_2_4_hidden_exposure` is the product-facing Block 2.4 contract (§2.4.1); populated when `build_portfolio_xray_v2` runs. It reads only `block_2_1_asset_allocation`, `block_2_2_portfolio_metrics`, and `block_2_3_factor_exposure`; it must not change those structures, read generated files, optimize, generate candidates, run Stress Lab, or recalculate factor models. Older on-disk files may omit the key until re-materialized. Legacy `sections.hidden_risk_detector` remains for golden tests and formatters.

`block_2_5_risk_budget_view` is the product-facing Block 2.5 contract (§2.5.1). It compares capital weights to RC_vol (`risk_contribution_pct`) and taxonomy risk-budget buckets. It is populated when `build_portfolio_xray_v2` runs ([Block 2.5 Risk Budget View ExecPlan](../exec_plans/2026-05-26_block_2_5_risk_budget_view_plan.md) **Completed** 2026-05-26). It must not recompute covariance, read `stress_report` for core fields, or include stress PnL in the product block. Older on-disk files may omit the key until re-materialized. Legacy `sections.risk_budget_view` remains for golden tests and formatters (may still include stress loss fields).

The current implementation may include additional fields for backward compatibility. New fields should be additive unless a future migration is explicitly planned.

## Common Section Contract

Each section should make evidence quality visible. The target common fields are:

- `status`: `implemented`, `partial`, `degraded`, or `unavailable`
- `data_sources_used`: source artifact names or in-memory inputs used
- `method`: calculation or rule summary where applicable
- `frequency`: daily, monthly, weekly, mixed, or not applicable
- `window`: calculation window or lookback period
- `n_obs`: observation count where applicable
- `benchmark`: benchmark used where applicable
- `items`: diagnostic rows or metric objects
- `warnings`: data, coverage, method, or interpretation warnings
- `limitations`: concise known limits
- `confidence`: high, medium, low, or unavailable where a section summarizes evidence

Existing sections do not yet meet this full target. The active plan moves them toward this contract incrementally.

## Data and Cutoff Policy

All Portfolio X-Ray diagnostics must respect the analysis-effective date:

- `analysis_end` is the latest completed period allowed for diagnostics.
- Raw cached or exported input panels may contain later incomplete rows only when clearly treated as raw input evidence.
- Diagnostic consumers must not calculate from or disclose `data_end` later than `analysis_end` unless the field is explicitly named as raw/unfiltered cache coverage.
- Generated diagnostic sections must prefer analysis-effective panels filtered to rows `<= analysis_end`.

If an input has insufficient coverage, stale data, young assets, NaNs, benchmark mismatch, FX issues, or missing taxonomy, the affected section must disclose a warning rather than silently implying full confidence.

#### 2.1.1 Block 2.1 product contract (`block_2_1_asset_allocation`)

Status: **implemented** (ExecPlan [Block 2.1 Asset Allocation MVP](../exec_plans/2026-05-26_block_2_1_asset_allocation_plan.md) **Completed** Sessions 01–08, 2026-05-26). **Implementation:** `src/block_2_1_asset_allocation.py`, wired from `build_portfolio_xray_v2` in `src/portfolio_xray.py`. Acceptance: [Block 2.1 acceptance audit](../audits/2026-05-26_block_2_1_asset_allocation_acceptance_audit.md).

Purpose: give portfolio-first consumers a **stable, product-facing** capital-allocation answer without parsing heterogeneous `sections.asset_allocation.items[]`. Diagnostic-only: no optimization, candidate selection, mandate gates, or trade instructions.

**Artifact placement:** top-level key `block_2_1_asset_allocation` on `portfolio_xray.json` under the active output folder (portfolio-first: `{output_dir_final}/analysis_subject/portfolio_xray.json`). Do **not** introduce a separate `asset_allocation.json` in the six-file product bundle.

**Backward compatibility:** `sections.asset_allocation` (items model) and `legacy_summary.asset_allocation_summary` remain required and unchanged in shape until an explicit migration. Report formatters may continue to use the legacy section; UI/API should prefer `block_2_1_asset_allocation`. `legacy_summary` is not the Core MVP product surface and must not be used for mandate/pass-fail product conclusions.

**Weight basis:** positive capital weights from resolved `analysis_setup` / analyzed weights (same as `_positive_weights` in `src/portfolio_xray.py`). No ETF look-through. Weights sum to at most `1.0`; partial sums disclose cash remainder via Input Layer, not via substituting `cash_proxy_ticker`.

**Real cash:** user labels such as `Cash USD`, `CASH`, `Cash EUR` are portfolio holdings with synthetic taxonomy (`taxonomy_source`: `real_cash_synthetic_v1`). They must appear in `by_asset` and `by_currency`. They must **not** be mapped to `cash_proxy_ticker` (e.g. BIL) for allocation. See [input_assumptions_spec.md](input_assumptions_spec.md) real-cash rule.

**Core MVP breakdown dimensions** (product contract): `by_asset`, `by_asset_class`, `by_main_risk_factor`, `by_risk_role`, `by_region`, `by_currency`.

**Deferred from core product contract** (may remain in legacy `sections.asset_allocation` items only): `sector`, `subtype`, `thematic_tags`, `secondary_risk_factor`, `duration_bucket`, `credit_quality`, full taxonomy table export.

### 2.2 Portfolio Metrics / Risk Diagnostics

Target metrics:

- CAGR
- volatility
- Sharpe
- Sortino
- max drawdown
- time to recovery
- VaR / ES with explicit method, frequency, window, and confidence
- beta
- Treynor when supported
- skewness / kurtosis

## Confidence and Warnings Policy

Every section should avoid false confidence:

- If data is missing, young, sparse, stale, or inconsistent, state it.
- If taxonomy is incomplete, state what allocation/risk views are incomplete.
- If benchmark-relative metrics are calculated, name the benchmark.
- If a metric uses daily, monthly, weekly, or mixed frequency, disclose it.
- If a section is based on thresholds, show enough evidence to explain why it was or was not flagged.
- If signals conflict, show the conflict rather than collapsing it into a single confident label.
