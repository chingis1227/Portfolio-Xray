# Portfolio X-Ray Diagnostics Specification

Status: source-of-truth for section contracts, thresholds, and JSON fields (Block 2).

Layer navigation (sub-blocks 2.1â€“2.7, code map, orchestration):
[portfolio_xray_layer_spec.md](portfolio_xray_layer_spec.md).

This spec defines the intended contract for the current-portfolio diagnostic layer. It is diagnostic-only: it describes how the system should explain an existing `analysis_subject` before any candidate comparison, optimization decision, or action plan.

## Scope

The Portfolio X-Ray layer diagnoses the current `analysis_subject` in the portfolio-first workflow:

```text
analysis_subject
-> subject diagnostics / Portfolio X-Ray
-> candidate generation and comparison
-> decision package
```

The layer is generated by the report path and written as `portfolio_xray.json` under the active output folder, including `{output_dir_final}/analysis_subject/` when `run_report.py --materialize-analysis-subject` is used.

The layer covers seven sub-blocks:

1. Asset Allocation
2. Portfolio Metrics / Risk Diagnostics
3. Factor Exposure / Factor Sensitivity
4. Hidden Exposure / Hidden Risk Detector
5. Portfolio Archetype Classification
6. Risk Budget View
7. Portfolio Weakness Map

## Non-Goals

Portfolio X-Ray must not:

- optimize the portfolio
- select candidates
- change current or generated weights
- change mandate gates, stress pass/fail status, or release decisions
- produce binding investment recommendations
- replace `candidate_comparison.json`, `selection_decision.json`, `action_plan.json`, or `portfolio_health_score.json`

Diagnostic statements may identify risks, gaps, evidence, uncertainty, and possible review areas. They must not be worded as binding buy/sell/hold instructions.

## Primary Artifacts

Machine-readable output:

- `portfolio_xray.json`

Current and target report consumers:

- `report.txt`
- `commentary.txt`
- generated HTML report surfaces
- generated PDF-style report surfaces

Generated report surfaces should render the X-Ray as structured sections and tables. `portfolio_xray.json` remains the canonical generated source for X-Ray rendering.

The current implementation may include additional fields for backward compatibility. New fields should be additive unless a future migration is explicitly planned.

## Common Section Contract

Each section should make evidence quality visible. The target common fields are:

- `status`: `implemented`, `partial`, `degraded`, or `unavailable`
- `data_sources_used`: source artifact names or in-memory inputs used
- `method`: calculation or rule summary where applicable
- `frequency`: daily, monthly, weekly, mixed, or not applicable
- `window`: calculation window or lookback period
- `n_obs`: observation count where applicable
- `benchmark`: benchmark used where applicable
- `items`: diagnostic rows or metric objects
- `warnings`: data, coverage, method, or interpretation warnings
- `limitations`: concise known limits
- `confidence`: high, medium, low, or unavailable where a section summarizes evidence

Existing sections do not yet meet this full target. The active plan moves them toward this contract incrementally.

## Data and Cutoff Policy

All Portfolio X-Ray diagnostics must respect the analysis-effective date:

- `analysis_end` is the latest completed period allowed for diagnostics.
- Raw cached or exported input panels may contain later incomplete rows only when clearly treated as raw input evidence.
- Diagnostic consumers must not calculate from or disclose `data_end` later than `analysis_end` unless the field is explicitly named as raw/unfiltered cache coverage.
- Generated diagnostic sections must prefer analysis-effective panels filtered to rows `<= analysis_end`.

If an input has insufficient coverage, stale data, young assets, NaNs, benchmark mismatch, FX issues, or missing taxonomy, the affected section must disclose a warning rather than silently implying full confidence.
