# Portfolio X-Ray Diagnostics Specification

Status: Skeleton source-of-truth for the active Portfolio X-Ray deepening plan.

This spec defines the intended contract for the current-portfolio diagnostic layer. It is diagnostic-only: it describes how the system should explain an existing `analysis_subject` before any candidate comparison, optimization decision, or action plan.

## Scope

The Portfolio X-Ray layer diagnoses the current `analysis_subject` in the portfolio-first workflow:

```text
analysis_subject
-> subject diagnostics / Portfolio X-Ray
-> candidate generation and comparison
-> decision package
```

The layer is generated by the report path and written as `portfolio_xray.json` under the active output folder, including `{output_dir_final}/analysis_subject/` when `run_report.py --materialize-analysis-subject` is used.

The layer covers seven sub-blocks:

1. Asset Allocation
2. Portfolio Metrics / Risk Diagnostics
3. Factor Exposure / Factor Sensitivity
4. Hidden Exposure / Hidden Risk Detector
5. Portfolio Archetype Classification
6. Risk Budget View
7. Portfolio Weakness Map

## Non-Goals

Portfolio X-Ray must not:

- optimize the portfolio
- select candidates
- change current or generated weights
- change mandate gates, stress pass/fail status, or release decisions
- produce binding investment recommendations
- replace `candidate_comparison.json`, `selection_decision.json`, `action_plan.json`, or `portfolio_health_score.json`

Diagnostic statements may identify risks, gaps, evidence, uncertainty, and possible review areas. They must not be worded as binding buy/sell/hold instructions.

## Primary Artifacts

Machine-readable output:

- `portfolio_xray.json`

Current and target report consumers:

- `report.txt`
- `commentary.txt`
- generated HTML report surfaces
- generated PDF-style report surfaces

Generated report surfaces should render the X-Ray as structured sections and tables. `portfolio_xray.json` remains the canonical generated source for X-Ray rendering.

## Top-Level JSON Contract

`portfolio_xray.json` should keep a stable top-level shape:

```json
{
  "version": "portfolio_xray_v1",
  "diagnostic_only": true,
  "diagnostic_only_disclaimer": "...",
  "analysis_setup_summary": {},
  "thresholds": {},
  "sections": {},
  "legacy_summary": {}
}
```

The current implementation may include additional fields for backward compatibility. New fields should be additive unless a future migration is explicitly planned.

## Common Section Contract

Each section should make evidence quality visible. The target common fields are:

- `status`: `implemented`, `partial`, `degraded`, or `unavailable`
- `data_sources_used`: source artifact names or in-memory inputs used
- `method`: calculation or rule summary where applicable
- `frequency`: daily, monthly, weekly, mixed, or not applicable
- `window`: calculation window or lookback period
- `n_obs`: observation count where applicable
- `benchmark`: benchmark used where applicable
- `items`: diagnostic rows or metric objects
- `warnings`: data, coverage, method, or interpretation warnings
- `limitations`: concise known limits
- `confidence`: high, medium, low, or unavailable where a section summarizes evidence

Existing sections do not yet meet this full target. The active plan moves them toward this contract incrementally.

## Data and Cutoff Policy

All Portfolio X-Ray diagnostics must respect the analysis-effective date:

- `analysis_end` is the latest completed period allowed for diagnostics.
- Raw cached or exported input panels may contain later incomplete rows only when clearly treated as raw input evidence.
- Diagnostic consumers must not calculate from or disclose `data_end` later than `analysis_end` unless the field is explicitly named as raw/unfiltered cache coverage.
- Generated diagnostic sections must prefer analysis-effective panels filtered to rows `<= analysis_end`.

If an input has insufficient coverage, stale data, young assets, NaNs, benchmark mismatch, FX issues, or missing taxonomy, the affected section must disclose a warning rather than silently implying full confidence.

## Section Contracts

### 2.1 Asset Allocation

Question answered: What does the portfolio own, and where is capital visibly concentrated?

Target views:

- asset-level weights
- asset class allocation
- region allocation
- currency allocation
- sector allocation
- risk bucket allocation
- ETF/stock taxonomy allocation
- capital concentration diagnostics

Current status: partially implemented. Asset-level weights and some metadata/taxonomy summaries exist. Region, currency, sector, risk bucket, and concentration views need fuller coverage and warnings.

### 2.2 Portfolio Metrics / Risk Diagnostics

Question answered: How has the portfolio behaved historically, and what risk did it take?

Target metrics:

- CAGR
- volatility
- Sharpe
- Sortino
- max drawdown
- time to recovery
- VaR / ES with explicit method, frequency, window, and confidence
- beta
- Treynor when supported
- skewness / kurtosis
- rolling beta and rolling correlation summaries
- downside beta and upside beta
- concentration metrics
- liquidity diagnostics when supported by data

Current status: partially implemented. Core metrics exist; daily historical VaR/ES with `tail_risk` metadata (method, frequency, window, n_obs) is available when daily data loads. Portfolio window metrics include skewness/kurtosis (monthly log returns), downside/upside beta, `corr_base`, `metric_quality` (n_obs, frequency, benchmark, risk-free source, window, analysis_end), and analytics rolling beta/correlation summaries (36m/12m) when benchmark data is available.

Canonical formula ownership remains in [metrics_specification.md](metrics_specification.md). This spec owns how those metrics are grouped, disclosed, and interpreted in the X-Ray layer.

### 2.3 Factor Exposure / Factor Sensitivity

Question answered: Which factors is the portfolio sensitive to?

Target factor coverage:

- equity
- rates
- inflation
- credit
- USD
- commodities
- volatility / VIX
- growth-style or growth proxy factors
- PCA/common factor evidence where available

Current status: partially implemented. Supporting factor/stress/PCA artifacts exist, but X-Ray does not yet expose all evidence consistently. Kalman betas are read from `stress_report.factor_betas_kalman.latest` (legacy `latest_betas_capped` / `latest_betas` only as fallback).

Canonical factor behavior remains in [factor_diagnostics_spec.md](factor_diagnostics_spec.md) and stress behavior in [stress_testing_spec.md](stress_testing_spec.md).

### 2.4 Hidden Exposure / Hidden Risk Detector

Question answered: What risks are not obvious from headline allocation?

Target categories:

- hidden equity beta
- duration concentration
- credit concentration
- liquidity concentration
- correlation/common-factor concentration
- weak hedge behavior
- tail risk
- hidden macro/factor dependency

Each category should include positive evidence, below-threshold evidence where relevant, method, threshold, confidence, and warnings.

Target V2 item fields (per category):

- `type`: `hidden_risk_flag` when flagged, else `hidden_risk_assessment`
- `name` / `category`: stable category id
- `flagged`: boolean
- `assessment_status`: `flagged`, `below_threshold`, or `unavailable`
- `severity`: `high`, `medium`, `low`, or null when below threshold
- `fact`, `interpretation`, `next_test`, `limitation`
- `evidence`: metric values and threshold keys used
- `thresholds`: list of named threshold keys from `portfolio_xray.json.thresholds`

Section-level fields (Hidden Risk Detector):

- `confidence`: `high`, `medium`, `low`, or `unavailable`
- `evidence_count`: categories with evaluable evidence (not `unavailable`)
- `flagged_count`, `below_threshold_count`, `unavailable_count`

Categories (minimum):

- `hidden_equity_beta`
- `duration_concentration`
- `credit_concentration`
- `liquidity_concentration`
- `correlation_or_common_factor_concentration` (raw PCA PC1)
- `residual_pca_concentration` (residual PCA PC1 after factor removal)
- `weak_hedge_behavior` (hedge-labeled holdings + stress/OOS explainability when available)
- `tail_risk` (daily historical ES/VaR from `portfolio_analytics.tail_risk` when available)
- `stress_loss_contributor_concentration`
- `macro_factor_dependency` (dominant factor betas / variance shares)
- `single_asset_risk_concentration` (Top1 RC_vol; optional but recommended)

Current status: V2 implemented in Session 05 (2026-05-19). Rule-based thresholds remain diagnostic-only.

### 2.5 Portfolio Archetype Classification

Question answered: What kind of portfolio is this, with evidence and caveats?

Target archetypes (rule-based scorecard, diagnostic-only):

- Equity Growth Portfolio
- Balanced 60/40-like
- Duration-heavy Defensive
- Credit Carry Portfolio
- Inflation-sensitive
- Pseudo-diversified Portfolio
- Defensive Portfolio
- Concentrated-risk Portfolio
- Tail-risk Exposed Portfolio
- Cash-heavy Low-Risk

Target V2 item fields:

- `type`: `portfolio_archetype`
- `primary_archetype`, `secondary_archetype` (nullable)
- `confidence`: `low` / `medium` / `high` from strength gap and tension count
- `positive_evidence`: supporting signals for the primary label
- `negative_evidence`: counter-signals and tensions for the primary label (including weakness-map contradictions)
- `drivers`: legacy alias of top positive evidence lines (backward compatible)
- `archetype_scorecard`: per-archetype rows with `archetype`, `strength`, `fit` (`primary` / `secondary` / `below_threshold`), `positive_evidence`, `negative_evidence`
- `conflicting_signals`: rows when a plausible archetype coexists with adverse weakness-map or counter-allocation evidence; each row includes `archetype`, `tension`, `explanation`, `positive_evidence`, `negative_evidence`, optional `related_weakness`, `weakness_severity`
- `conflict_summary`: one English sentence when any tension is present

Section-level fields:

- `confidence`: mirrors item confidence when a single item is emitted
- `limitations`: diagnostic-only caveat and multi-label disclaimer

Scorecard rules use named thresholds from `portfolio_xray.json.thresholds` plus allocation, factor betas, RC_vol, PCA, tail risk, hidden-risk flags, and weakness-map rows. A primary label must not hide simultaneous inflation exposure and high inflation/rates weakness — that tension belongs in `negative_evidence` and `conflicting_signals`, not in silent omission.

Current status: V2 implemented in Session 07 (2026-05-20). Archetype is built after the weakness map so regime tensions can be disclosed.

### 2.6 Risk Budget View

Question answered: Which assets or factors consume portfolio risk relative to their capital weights?

Target comparisons:

- capital weight
- volatility risk contribution
- stress loss contribution
- factor risk contribution
- drawdown contribution if implemented
- asset-level contribution to total risk

Current status: partially implemented. Risk Budget View loads full `results_csv/rc_vol_10y.csv` evidence (5Y/3Y fallback) when available; `snapshot_10y.json.RC_asset` remains a display-oriented top-N subset for snapshot/report tables only.

Canonical RC formula ownership remains in [metrics_specification.md](metrics_specification.md). This spec owns the risk-budget display contract and completeness expectations.

### 2.7 Portfolio Weakness Map

Question answered: Under which scenarios or regimes is the portfolio vulnerable, and why?

Target vulnerability rows:

- recession
- inflation shock
- rates-up shock
- credit spread widening
- liquidity shock
- USD shock
- equity crash
- commodity shock
- volatility spike
- crypto shock only when relevant exposure exists

Each row should separate:

- `exposure_present` — portfolio has a channel to the risk (taxonomy, factor beta, or mapped scenario relevance)
- `adverse_evidence` — available diagnostics crossed configured moderate/high thresholds (not “risk absent”)
- `severity` — `low` / `medium` / `high` from aggregated evidence
- `confidence` — `low` / `medium` / `high` from evidence count and missing inputs
- `scenario_coverage` — `mapped_scenarios`, `scenarios_present`, `scenarios_missing`
- `top_asset_loss_drivers` — up to three tickers with worst `pnl_by_asset_pct` from mapped stress scenarios
- `top_factor_drivers` — factor beta and/or `pnl_by_factor_pct` contributors for the weakness
- `evidence` — structured evidence rows (stress, factor, drawdown, tail, taxonomy)
- `missing_inputs` — per-row gaps (e.g. mapped scenario absent from `scenario_results`)
- `interpretation` — English diagnostic sentence

`crypto_shock` is emitted only when crypto exposure is present (`asset_class=crypto` or `main_risk_factor=crypto_beta`).

Low severity with `adverse_evidence=false` means thresholds were not crossed, not that the risk is impossible.

Current status: implemented (Weakness Map V2, Session 06, 2026-05-19).

## Confidence and Warnings Policy

Every section should avoid false confidence:

- If data is missing, young, sparse, stale, or inconsistent, state it.
- If taxonomy is incomplete, state what allocation/risk views are incomplete.
- If benchmark-relative metrics are calculated, name the benchmark.
- If a metric uses daily, monthly, weekly, or mixed frequency, disclose it.
- If a section is based on thresholds, show enough evidence to explain why it was or was not flagged.
- If signals conflict, show the conflict rather than collapsing it into a single confident label.

## Verification

Minimum documentation verification:

```bash
python scripts/verify_docs.py
```

Expected later implementation checks:

- focused X-Ray tests under `tests/test_portfolio_xray.py`
- focused metrics/frequency tests
- focused data-cutoff tests
- generated artifact review of `{output_dir_final}/analysis_subject/portfolio_xray.json`
- generated report review for `report.txt`, `commentary.txt`, HTML, and PDF-style surfaces

Acceptance criteria:

- no diagnostic section calculates from or claims analysis-effective dates after `analysis_end`
- risk budget includes all positive-weight holdings when full evidence exists
- factor exposure uses available Kalman evidence
- tail-risk fields disclose method, frequency, and window
- hidden risk, archetype, and weakness outputs show evidence, confidence, and caveats
- report surfaces are understandable without reading raw JSON
